# Roadmap

Eyepiece ports 18 Telescope entry types. Most screens reuse the same primitives — a paginated table on the index, a JSON-ish detail panel on the show. Only a handful diverge meaningfully. The plan reflects that.

## Phase 1 — shared primitives

The design handover in `design_handoff_telescope_requests/` defines the visual + interaction spec for the Requests screen — and via it, for every list screen. Build these to that spec:

- **App chrome** — `<Header>` (logo + brand + live pill + icon buttons), `<Sidebar>` (Watchers / Data / Diagnostics sections, nav items with count badges, active state).
- **`<EntryTable>`** — sticky header, fixed-width grid columns, sortable + selectable rows, per-row badges (Verb, Status), threshold-colored numeric cells. Empty state.
- **`<EntryDetailDrawer>`** — slide-over panel (not a separate route). Backdrop + fixed-right 520px aside. Open driven by `selectedId` URL param. Each entry type fills the body; the head (eyebrow / title / close) is shared.
- **`<SummaryStatCards>`** — Avg / p95 / Errors, computed over the filtered set.
- **Filter form** — search input, multi-select verb chips, multi-select status chips with counts, single-select duration chips, Clear button visible only when filters are active.
- **Saved views row** — pills bound to URL state; preset + persistable user views.
- **Live polling + Pause** — header pill driven by interval state, pause/resume button.
- **URL state** — search, verbs, statuses, minDuration, selected view, selected entry id all in the querystring (shareable, survives reload).
- **Column chooser** — toggle visibility; Verb + Path mandatory.
- **Cursor pagination** — `before` cursor (Telescope's native paging).
- **Design tokens module** — colors, typography (IBM Plex Sans + Mono), radii, verb/status palettes — pulled from `design_handoff_telescope_requests/README.md` into `resources/js/lib/design-tokens.ts` (or CSS vars in `styles.css`). Light theme default, dark theme behind a toggle.
- **Avatar** — initials + name-hashed color, guest/API variants.
- **Response typing pattern** — co-locate the list/detail types in `lib/api.ts` per entry type, generated by reading `/mnt/projects/telescope/src/Watchers/<Type>Watcher.php`.

The Requests screen is the reference: when these land, Requests is pixel-accurate to `screenshots/01-list.png` + `02-detail-panel.png` and other list screens drop in with just column config + detail body.

## Phase 2 — port the bulk of the screens

Each one is "type the response shape, plug into the primitives." Order by user value:

1. Requests detail
2. Exceptions (list + detail)
3. Queries
4. Logs
5. Jobs
6. Commands
7. Mail (list — preview is Phase 3)
8. Cache
9. HTTP Client
10. Notifications
11. Models
12. Events
13. Gates
14. Redis
15. Views
16. Schedule
17. Batches

Cross-check field names against `/mnt/projects/telescope/src/Watchers/<Type>Watcher.php` (writes the `content` blob) and `/mnt/projects/telescope/resources/js/screens/<type>/` (what the original UI rendered).

## Phase 3 — the screens that don't fit the mold

- **Mail preview** — HTML, not JSON. Render via `<iframe src="/telescope-api/mail/{id}/preview">`. `.eml` download is a separate link to `/telescope-api/mail/{id}/download`.
- **Exceptions stack trace** — `content` contains pre-rendered HTML for the highlighted frames. Render with `dangerouslySetInnerHTML` — Telescope-generated, trusted.
- **Dumps** — no detail view; the list IS the screen.
- **Monitored Tags** — CRUD against `GET/POST /telescope-api/monitored-tags`. Not paginated; small typed form.

## After all of the above

- `composer install`, `npm install`, smoke test via `composer run serve`.
- Pest test (view override) and PHPStan green.
- Design handover from `design_handoff_telescope_requests/` informs the styling of the Phase 1 primitives — `styles.css` gets replaced when the design lands.
